Agentic AI Architecture for Automated Candidate Placement

This document outlines a fully automated, agent-driven architecture for a complete staffing lifecycle, spanning Candidate Intake (ATS), Job Requisition (VMS), and Automated Placement. The design utilizes the Model Context Protocol (MCP) to decouple reasoning agents from external tools and databases, and incorporates self-healing loops to ensure resilient operation.

1. System Overview

The architecture is built around a central Orchestrator Agent that delegates specialized tasks to a fleet of autonomous sub-agents. Each sub-agent communicates with external systems (Databases, Email, VMS portals) exclusively through standard MCP Servers.

2. Model Context Protocol (MCP) Server Ecosystem

Instead of hardcoding API integrations into the agents, we expose capabilities via MCP servers. This allows seamless agent capability expansion.

  1. ATS MCP Server:
    • Backend: Postgres/Supabase with vector(768).
    • Tools Exposed: add_candidate(resume_data), update_candidate_status(id, status), semantic_search_candidates(job_description).
  2. VMS Integrations MCP Server:
    • Backend: Headless browser automation (Playwright/Puppeteer) and API clients.
    • Tools Exposed: fetch_new_requisitions(), submit_candidate(job_id, candidate_profile), check_submission_status(submission_id).
  3. Communication MCP Server:
    • Backend: Microsoft Graph API, Twilio, WhatsApp.
    • Tools Exposed: send_email(to, subject, body), send_sms(to, message), read_inbox().

3. Specialized Agent Fleet

3.1. Intake & ATS Agent (The Recruiter)

  • Role: Automatically processes inbound candidate resumes.
  • Workflow:
    1. Detects new resume via Communication MCP.
    2. Extracts skills, experience, and generates a 768-dimensional embedding.
    3. Uses the ATS MCP to create/update the candidate profile.

3.2. VMS & Requisition Agent (The Account Manager)

  • Role: Continuously polls VMS platforms for new job openings.
  • Workflow:
    1. Uses VMS MCP to fetch new jobs.
    2. Parses job descriptions and stores them in the vector database.
    3. Alerts the Orchestrator that a new requisition is open.

3.3. Matching & Outreach Agent (The Sourcer)

  • Role: Finds the right candidate for the job and gains their consent.
  • Workflow:
    1. Triggered by a new job requisition.
    2. Calls semantic_search_candidates on ATS MCP to find the top matches.
    3. Uses Communication MCP to send hyper-personalized emails/SMS asking for RTR.
    4. Waits for and parses reply. If "Yes", notifies Orchestrator.

3.4. Submission Agent (The Coordinator)

  • Role: Formats candidate data and officially submits them to the VMS.
  • Workflow:
    1. Retrieves candidate data and explicit consent.
    2. Uses VMS MCP submit_candidate tool.
    3. Logs the submission ID in the ATS.

4. Self-Healing & Resilience Mechanisms

To ensure "all agents run successfully" without human intervention, a dedicated Self-Healing Loop is essential.

  1. VMS Schema Drift Detection:
    • Failure: VMS Agent fails to submit because the VMS added a new required field (e.g., "Visa Status").
    • Healing: The Submission Agent catches the error, spawns a temporary Outreach Agent to email the candidate: "We need your Visa Status". Once replied, the Submission Agent retries.
  2. Communication Failures:
    • Failure: Email bounces or API token expires.
    • Healing: The Monitor Agent triggers a token refresh sequence and instructs the Outreach Agent to fallback to SMS.
  3. Infinite Loop Prevention (Circuit Breakers):
    • Agents maintain a state token. After 3 failed attempts, the Orchestrator places the task in a "Human Review Queue".

5. Fully Automated Lifecycle Example

  1. Event: VMS Agent detects a "Senior Cloud Architect" role.
  2. Matching: Matching Agent queries ATS, finds Candidate A (95% match).
  3. Outreach: Matching Agent emails Candidate A.
  4. Response: Candidate A replies "I'm interested. Please submit me."
  5. Submission: Orchestrator triggers Submission Agent.
  6. Self-Healing Trigger: Profile is missing the required "Expected Bill Rate".
  7. Correction: Outreach Agent texts Candidate A asking for their rate. Candidate replies "85/hr".
  8. Finalization: ATS is updated, Submission Agent successfully submits to VMS. Orchestrator logs success.

1. Orchestrator API & Task Queuing

What should we do?

We need to receive resumes and large candidate datasets from the frontend instantly, without making the user wait for complex AI processing (which can take 10-30 seconds per document). We need a scalable way to handle traffic spikes.

How did we achieve it?

We built the FastAPI Orchestrator deployed on Azure Container Apps. When a user uploads a resume via `candidates.html`, the API accepts the payload, quickly uses Gemini 2.0 Flash to extract structural text, and immediately enqueues a background task (e.g., `ingest_resume`) into our Azure PostgreSQL Database. The user gets an instant success response while the heavy lifting happens behind the scenes.

🤖

2. Specialized Micro-Agents

What should we do?

Instead of one monolithic AI trying to do everything (parse, match, submit, track), we need isolated contexts so that if a vendor management system (VMS) changes, it doesn't break our applicant tracking system (ATS) logic.

How did we achieve it?

We adopted a decoupled micro-agent fleet architecture. All agents inherit from a common `base_agent.py` which provides the loop to pull work from the PostgreSQL database queue. We created specialized workers:
• ATSAgent: Only parses and syncs candidate profiles.
• VMSAgent: Only pulls open jobs and pushes submissions to vendor portals.
• Submission/Sourcing Agents: Handle niche outreach tasks concurrently.

🎯

3. AI Semantic Matching

What should we do?

Traditional keyword search is inadequate for modern recruiting. We need to match candidates to open jobs based on semantic meaning, skills context, and conceptual overlap, rather than exact text matches.

How did we achieve it?

The MatchingAgent utilizes the `vector` extension in our Azure PostgreSQL database (`pgvector`). When candidates are ingested, their skills and experience are converted into embeddings via Gemini/embedding models. When a new job opens, the agent runs a cosine similarity vector search across the database to identify the top N candidates conceptually, scoring them autonomously.

🏥

4. Self-Healing Architecture

What should we do?

AI models occasionally hallucinate, APIs timeout, or parsing fails on complex PDFs. We cannot afford to silently drop tasks or have deadlocks in the queue. We need resilience.

How did we achieve it?

We created a dedicated SelfHealingAgent. Instead of processing raw jobs, it continuously sweeps the task queue for tasks that are marked "failed" or have been "pending" too long. It automatically requeues transient failures. If a task fails repeatedly, it marks it for Human Review and uses Microsoft Graph API to send an email alert, ensuring zero dropped candidates.

☁️

5. Azure Container Scaling

What should we do?

We need the infrastructure to be cost-effective during quiet hours but capable of massively scaling horizontally when 1,000 resumes drop during a career fair.

How did we achieve it?

By deploying on Azure Container Apps, we achieve independent horizontal scaling. The Orchestrator container can scale to handle web traffic, while the Agent Fleet containers can scale out based on queue depth. They use Azure Key Vault to securely inject credentials across all spun-up replicas without hardcoding API keys.

🔒

6. Secure Data Isolation

What should we do?

Resumes contain highly sensitive Personally Identifiable Information (PII). We must ensure the database is absolutely protected from public internet exposure while remaining accessible to our agents.

How did we achieve it?

We use Azure Database for PostgreSQL Flexible Server configured with `AllowAllAzureIPs`. It sits within an isolated subnet. The only entities that can read/write candidate data are the internal container apps hosting our agents, creating an air-gapped environment from the public-facing Firebase frontend.
